# Product Specs

Every feature starts here. A spec defines what users can do and what "done"
looks like — not how the code is structured. Acceptance scenarios and tests must
be grounded in a spec in this directory.

## Spec template

```markdown
# Product Spec: <Feature name>

**Status:** Draft / Active / Done
**Feature file:** <path to the acceptance scenarios>

## Goal

One paragraph: who needs this and why.

## Scope

### In scope
- <capability the user gains>

### Out of scope
- <adjacent capability deliberately excluded>

## Acceptance criteria

### AC-1: <criterion name>

Given / when / then prose precise enough to write a scenario from.

## Open questions

- <decision pending, with the current default>
```
